# 00 — How Flutter Apps Work (for a Java / Spring Boot developer)

> Read this once. After this, every other doc in this folder will make sense without re-explaining the basics.

You know Spring Boot. You know "annotate a class with `@Component`, the container wires it, your `main` method boots Tomcat, your `@RestController` handles HTTP, your `@Repository` talks to JPA which talks to the DB." That mental model maps almost 1-to-1 onto what we built — only the names are different.

This doc gives you those name mappings, plus the few things Flutter does that Spring doesn't (the "widget tree" and the "build" cycle), so you can read any `.dart` file in the codebase without surprise.

---

## 1. The 60-second mental model

| Spring Boot world | Flutter / Dart world | In this app |
|---|---|---|
| JVM | Dart VM (debug) / native ARM machine code (release) | Compiled by the Flutter toolchain |
| `public static void main(String[] args)` | `Future<void> main() async { ... }` | `lib/main.dart` |
| `SpringApplication.run(App.class, args)` | `runApp(EndlessApp())` | end of `main.dart` |
| `@Configuration` / `@Bean` | `Provider<T>((ref) => ...)` | `core/services/service_providers.dart` |
| `@Component`, `@Service` | `Provider`, `AsyncNotifierProvider` | every `*_provider.dart` |
| `@Autowired` constructor injection | `ref.watch(someProvider)` | inside every screen / notifier |
| `@RestController` returning a view | `Widget build(BuildContext ctx)` returning a tree of widgets | every `*_screen.dart` |
| `@Repository` interface | `abstract class FooRepository` | `domain/repositories/*.dart` |
| `@Repository` JPA impl | `class FooRepositoryImpl extends FooRepository` | `data/repositories/*_impl.dart` |
| JPA `@Entity` | Isar `@collection` class (the model) + a plain Dart entity | `data/models/*_model.dart` + `domain/entities/*.dart` |
| H2 / embedded DB | **Isar** — embedded NoSQL DB on the device | `core/database/isar_service.dart` |
| Service layer method (`OrderService.placeOrder`) | **Use case** class with a single `call()` method | `domain/use_cases/*.dart` |
| Thymeleaf / JSP rendered view | **Widget tree** rebuilt reactively when state changes | every `Widget build()` |
| Tomcat thread pool | Dart's single-thread **event loop** (like Node.js) | implicit — see §10 |
| `CompletableFuture<T>` | `Future<T>` | everywhere `async`/`await` appears |
| `Flux<T>` / reactive stream | `Stream<T>` | `AppLogger.I.stream`, Isar `.watch()` |
| `application.yml` | `pubspec.yaml` | `app/pubspec.yaml` |
| Maven / Gradle | the `flutter` and `dart pub` CLI | run `flutter pub get` instead of `mvn install` |

If you stop reading here, you can already place every file in the project. The rest fills in *why* the names are different and what details bite you.

## 2. What actually runs on the phone

When the user taps the **Endless** icon on their phone:

1. **Android OS** launches the app process and calls into our `MainActivity` (auto-generated by Flutter — we don't touch it). It's a Kotlin class roughly equivalent to a Spring Boot `main` that just boots a `FlutterEngine`.
2. The **Flutter engine** (a C++ runtime shipped inside the APK) starts the **Dart VM** (in debug mode) or executes pre-compiled **ARM machine code** (in release mode — our `v1.0.0` build).
3. The engine calls our Dart entry point: `main()` in `lib/main.dart`.
4. From that moment, everything is **our Dart code**. The screen pixels are drawn by **Skia** (Google's 2D graphics library, also bundled in the APK). Flutter does not use the native Android widgets — every button you see is drawn by Skia using shapes we describe in Dart.

So Flutter is closer to a **game engine** than to a typical native app: we describe what should be on screen, and Skia paints it. That's why a Flutter app looks identical on Android and iOS — the OS draws nothing, we draw everything.

**Why this matters for you:** there is no XML layout file, no Storyboard, no JSX. The UI is *only* what `Widget build()` methods return. To understand a screen, you read one `build()` method top-to-bottom.

## 3. The boot sequence — what runs first

In Spring Boot:

```java
public static void main(String[] args) {
    SpringApplication.run(EndlessApp.class, args);
}
```

…boots the IoC container, scans `@Component`s, starts Tomcat, then returns. Your `@RestController`s sit idle waiting for HTTP.

In our app, `lib/main.dart`:

```dart
Future<void> main() async {
  await runZonedGuarded(_bootstrap, (error, stack) {
    AppLogger.I.error('zone', 'uncaught zone error', error: error, stack: stack);
  });
}
```

Boot sequence step-by-step:

1. `main()` — Dart's entry point (just like Java's `public static void main`).
2. `runZonedGuarded(_bootstrap, ...)` — wrap everything inside `_bootstrap()` in a *zone* (think: a custom `UncaughtExceptionHandler` thread group in Java). Any async error that escapes goes to our logger.
3. `_bootstrap()`:
   - `WidgetsFlutterBinding.ensureInitialized()` — tells the Flutter engine "we want to call platform APIs (file system, notifications) before `runApp`." Without this line you get a crash if you touch any plugin before the UI starts.
   - `AppLogger.I.init()` — open our log file (`<app_docs>/logs/app.log`).
   - `FlutterError.onError = ...` — install a global handler for **synchronous** UI errors → logger.
   - `PlatformDispatcher.instance.onError = ...` — global handler for **async/native** errors → logger. Together with `runZonedGuarded`, this is our complete crash-capture net.
   - Initialize the three platform-touching services in order: **Isar** (DB), **Notifications**, **Alarm**. Each wrapped in `_guard()` so one failing service doesn't kill app start.
   - `runApp(ProviderScope(overrides: [...], child: const EndlessApp()))` — hands control to Flutter. From this point the app is alive and reactive.

4. `runApp` builds the widget tree. The first widget is `ProviderScope` (Riverpod's IoC container — analogous to the Spring `ApplicationContext`). Inside it sits `EndlessApp`, which sits inside `MaterialApp`, which renders `SplashScreen` as its first screen.

5. `SplashScreen` shows the logo for ~1.5s, then routes to either onboarding (first run) or `HomeShell` (returning user).

**Key idea:** in Spring Boot, `main` returns and the app idles waiting for requests. In Flutter, `main` returns and the **widget tree** stays alive, listening for user input (taps, swipes) and rebuilding itself when state changes.

## 4. Dart syntax cheatsheet for a Java dev

Dart is very close to Java. You can read 90% of Dart code with no preparation. The differences:

| Java | Dart | Notes |
|---|---|---|
| `String s = "hi";` | `final s = 'hi';` or `var s = 'hi';` | `final` ≈ Java `final`. `var` is type-inferred (like Java's `var` in Java 10+) |
| `final String s;` | `late final String s;` | `late` defers initialization until first use. Like a non-null `@Nullable` you promise to set before reading. |
| `int x = 5;` immutable variable | `const x = 5;` | `const` = compile-time constant. Stronger than `final`. |
| `class Foo { Foo(this.x); int x; }` | `class Foo { Foo(this.x); int x; }` | Same syntax. Dart also supports `Foo({required this.x})` for named parameters. |
| `void doIt(String name)` | `void doIt(String name)` | Identical. Dart adds named params: `void doIt({required String name})` called as `doIt(name: 'bob')`. |
| `Optional<T>` / nullable types | `T?` (nullable) vs `T` (non-null) | Built into the language. `String?` can be null, `String` cannot. Compiler enforces. |
| `obj?.foo()` | `obj?.foo()` | Same null-safe call. |
| `obj!.foo()` | `obj!.foo()` | `!` is "I promise this is not null, crash if it is" — like Kotlin's `!!`. |
| `if (a != null) doIt(a)` | `if (a != null) doIt(a)` | Same. Dart also smart-casts: inside the if, `a` is typed as non-null. |
| `interface Foo {}` | `abstract class Foo {}` | Dart uses abstract classes as interfaces. |
| `class Bar implements Foo` | `class Bar implements Foo` | Same. |
| lambda `(x) -> x + 1` | `(x) => x + 1` | Arrow fn. Block form: `(x) { return x + 1; }`. |
| `List<String> xs = new ArrayList<>();` | `final xs = <String>[];` | List literals. `<String>[]` = empty `List<String>`. |
| `Map<String, Integer> m = ...;` | `final m = <String, int>{};` | Same idea. |
| `CompletableFuture<String>` | `Future<String>` | See §10. |
| `Stream<String>` | `Stream<String>` | Reactive stream. Subscribe with `.listen(...)`. |
| `enum Color { RED, GREEN }` | `enum Color { red, green }` | Same. Switch supports `switch (c) { Color.red => '...', ... }` (Dart 3 pattern matching). |
| `record Point(int x, int y)` (Java 16+) | `(int x, int y)` — record/tuple | Used in our logs screen for `('All', null)`. Access via `.x`, `.y` or `.$1`, `.$2`. |

**Three Dart-isms with no direct Java equivalent:**

1. **Named & required parameters:** `Widget build({required String title, int? count})`. Caller writes `build(title: 'X')`. Used everywhere in Flutter because widgets have many optional config knobs.

2. **Cascade operator `..`:** `buf..write('a')..write('b')` is `buf.write('a'); buf.write('b');` — chains method calls on the *same* object. (Java has `StringBuilder.append('a').append('b')` returning `this`; Dart works on any method.)

3. **Spread `...` and null-safe spread `...?`:**
   ```dart
   [1, 2, ...otherList]            // splice another list in
   [1, 2, ...?maybeNullList]       // splice if non-null, else nothing
   ```
   You'll see this in `notes_provider.dart` line 43 when we rebuild the list with the new note inserted.

That's enough. If a piece of syntax surprises you, search the **Dart language tour** — it's short.

## 5. Widgets: the unit of UI

In Spring, your "view layer" is a Thymeleaf template — HTML with `${name}` placeholders. The template is *separate* from the controller. The data flow is: controller → model → template renders → HTML.

In Flutter, there are no templates. The UI is built **out of widgets**, which are Dart classes. Every visual thing — a button, a column, padding, even color — is a widget. You compose them like Lego.

```dart
class HelloCard extends StatelessWidget {
  final String name;
  const HelloCard({required this.name});

  @override
  Widget build(BuildContext context) {
    return Container(
      padding: const EdgeInsets.all(16),
      decoration: BoxDecoration(color: Colors.blue),
      child: Text('Hello, $name'),
    );
  }
}
```

This is a class. To put it on screen: `HelloCard(name: 'Ashish')` somewhere inside another widget's `build()`.

**Two flavours of widget:**

- **`StatelessWidget`** — pure function of its inputs. Given the same constructor args, always renders the same thing. (Like an immutable Spring DTO.)
- **`StatefulWidget`** — has internal mutable state. Splits into two classes: the widget (immutable config) + a `State<T>` object (mutable). Use this when the widget tracks something itself: text field contents, current tab index, animation progress. Examples in our code: `HomeShell` (tracks `_currentIndex`), `NotesListScreen` (tracks `_isSearching`).

**The build cycle:**

1. Something changes (user taps, state updates, timer fires).
2. Flutter marks the affected widget "dirty."
3. On the next frame (~16ms later), Flutter calls `build()` on the dirty widget.
4. `build()` returns a *new* widget tree.
5. Flutter compares (diffs) the new tree against the previous one — like React's virtual DOM diff.
6. Only the differences are passed to Skia to repaint.

You **don't manipulate widgets after creating them**. You don't `button.setText("X")`. You set state, the widget rebuilds, the new tree contains the new text. This is the single biggest mental shift from "imperative" UI frameworks (Swing, Android XML, vanilla JS).

**Why so many widgets in one `build()`?** Because composition is the only tool. Want padding around a button? Wrap it in `Padding(child: button)`. Want it centered? Wrap in `Center(child: ...)`. The tree gets deep but each piece does exactly one thing.

## 6. State: ephemeral vs application

Two kinds of state in any app:

| Kind | Scope | Solution in this app |
|---|---|---|
| **Ephemeral / UI** — "is the search bar open?", "what's typed in the field?", "which tab am I on?" | One widget | `StatefulWidget` + `setState()` |
| **Application** — "list of notes," "monthly summary," "is user authenticated?" | Many widgets across screens | **Riverpod providers** (next section) |

Spring analogy: ephemeral state is like a `@RequestScoped` bean — one HTTP request, dies after. Application state is like a `@Singleton` `@Service`.

`setState()` is simple: call it inside a `StatefulWidget` and Flutter schedules a rebuild of that widget.

```dart
void _toggle() {
  setState(() {
    _isOpen = !_isOpen;       // Mutate the field
  });                          // Flutter sees this, rebuilds the widget
}
```

You **must** wrap the mutation in `setState(() { ... })`. If you mutate without it, the field changes but the UI doesn't redraw — a common bug.

## 7. Riverpod: the DI container (= Spring `ApplicationContext`)

This is the part that maps most directly to Spring, but with different ceremony. Read this section twice.

### What Riverpod is

A **dependency-injection container** with **observable state**. You declare a "provider" (Spring `@Bean`); anywhere in the widget tree, you `ref.watch(theProvider)` (Spring `@Autowired`) to get the value *and* automatically rebuild your widget when that value changes.

There's no annotation scanning — providers are global top-level variables declared with `final`. Their *consumers* (widgets) explicitly read them.

### Provider flavours we use

| Provider | Spring equivalent | When to use |
|---|---|---|
| `Provider<T>` | `@Bean` returning a singleton, immutable | Stateless services: `noteRepositoryProvider`, `appLoggerProvider` |
| `StateProvider<T>` | A mutable `@Bean` holding a single value | Tiny mutable state shared across widgets: `noteSearchQueryProvider` (the current search string) |
| `FutureProvider<T>` | A `@Bean` of `CompletableFuture<T>` | Async one-shot data: `monthlySummaryProvider` |
| `AsyncNotifierProvider<N, T>` | A `@Service` with mutable state, async load + CRUD methods | The main pattern: `notesProvider`, `tasksProvider`, etc. |

### Example: provider + consumer

Declare (in `notes_provider.dart`):

```dart
final notesProvider = AsyncNotifierProvider<NotesNotifier, List<Note>>(NotesNotifier.new);

class NotesNotifier extends AsyncNotifier<List<Note>> {
  late NoteRepository _repository;

  @override
  Future<List<Note>> build() async {
    _repository = ref.watch(noteRepositoryProvider);  // DI: get the repo
    return _repository.getAllNotes();                  // initial load
  }

  Future<void> create(Note note) async { ... }
  Future<void> delete(int id) async { ... }
}
```

Consume (in `notes_list_screen.dart`):

```dart
class NotesListScreen extends ConsumerStatefulWidget { ... }

class _NotesListScreenState extends ConsumerState<NotesListScreen> {
  @override
  Widget build(BuildContext context) {
    final notesAsync = ref.watch(notesProvider);    // re-runs build() when notes change

    return notesAsync.when(
      loading: () => CircularProgressIndicator(),
      error:   (e, st) => Text('Error: $e'),
      data:    (notes) => ListView(...),
    );
  }
}
```

Three things to notice:

1. The widget class is `ConsumerStatefulWidget`, not plain `StatefulWidget`. That's how the widget gets the magic `ref`. There's also `ConsumerWidget` for stateless consumers.
2. `ref.watch` **subscribes** — the widget rebuilds when the provider's value changes. `ref.read` is the same but **doesn't subscribe** (use inside button callbacks: "do this once, don't rebuild me when it changes").
3. `AsyncValue<T>.when(...)` is a built-in pattern matcher for "loading / error / data". Equivalent to a `switch` on the three states of a `CompletableFuture`.

### The `ref` everywhere

`ref` is the link to the IoC container. Inside notifiers, providers, and consumer widgets you always have a `ref`. It has three core methods:

- `ref.watch(p)` — get value and subscribe
- `ref.read(p)` — get value once, no subscription (use in callbacks)
- `ref.invalidate(p)` / `ref.invalidateSelf()` — "the underlying data changed; please re-run `build()`"

The whole tree of providers + `ProviderScope` at the root = the IoC container.

### Overriding providers (testing & startup)

In `main.dart`:

```dart
runApp(
  ProviderScope(
    overrides: [
      isarServiceProvider.overrideWithValue(isarService),
      ...
    ],
    child: const EndlessApp(),
  ),
);
```

This is Riverpod's version of Spring's `@TestConfiguration` + `@MockBean`. We use it here because the Isar / notification / alarm services need an `await init()` *before* the app starts, and `ProviderScope` can't `await` inside its constructor. So we init eagerly in `main`, then override.

## 8. Clean Architecture — three layers per feature

Every feature (notes, tasks, reminders, alarms, money) is split into three folders:

```
features/notes/
├── domain/          ← pure Dart, no Flutter, no DB
│   ├── entities/        ← Plain Old Dart Object (POJO equivalent)
│   ├── repositories/    ← Abstract interface
│   └── use_cases/       ← Service-layer method, one class per action
├── data/            ← knows about the DB
│   ├── models/          ← Isar annotated model (= JPA @Entity)
│   ├── datasources/     ← Direct Isar queries (= JpaRepository methods)
│   └── repositories/    ← Implements domain interface using datasource
└── presentation/    ← Flutter widgets
    ├── providers/       ← Riverpod glue (= @Service holding state)
    ├── screens/         ← Full-screen widgets
    └── widgets/         ← Reusable sub-widgets
```

Dependency rule: **outer layers depend on inner, never reverse.**

- `domain` knows nothing about Isar or Flutter.
- `data` imports `domain` to implement its interfaces.
- `presentation` imports `domain` (uses entities + use cases) and Riverpod providers wire in `data` implementations.

**Why bother?** Because:
- We can swap Isar for SQLite without touching `domain` or `presentation`.
- Use cases are tiny and unit-testable with no Flutter/DB dependencies.
- A new dev (you) can read a feature top-down: "what does this feature *do*" (`domain`) → "how is it stored" (`data`) → "what does it look like" (`presentation`).

### Spring analogy table

| In our app | Spring Boot equivalent |
|---|---|
| `domain/entities/note.dart` — `class Note { final String body; ... }` | A POJO / record class, no JPA annotations |
| `domain/repositories/note_repository.dart` — `abstract class NoteRepository { Future<List<Note>> getAllNotes(); }` | `interface NoteRepository { List<Note> findAll(); }` |
| `domain/use_cases/create_note_use_case.dart` — `class CreateNoteUseCase { Future<int> call(Note n) => _repo.create(n); }` | A `@Service` method `NoteService.create(Note n)` |
| `data/models/note_model.dart` — `@collection class NoteModel { Id id; ... }` | `@Entity class NoteEntity { @Id Long id; ... }` (JPA) |
| `data/datasources/note_local_datasource.dart` — `class NoteLocalDatasource { final Isar isar; ... }` | `JpaRepository<NoteEntity, Long>` or a hand-rolled DAO |
| `data/repositories/note_repository_impl.dart` — `class NoteRepositoryImpl implements NoteRepository` | The concrete `@Repository` class implementing the interface |
| `presentation/providers/notes_provider.dart` — `notesProvider` | A `@Component` `NotesViewModel` injected into the controller |
| `presentation/screens/notes_list_screen.dart` | A Thymeleaf view + Controller fused into one |

### Why have *both* `Note` (entity) and `NoteModel` (data model)?

Because `Note` is what the rest of the app uses — pure Dart, no Isar awareness. `NoteModel` exists only to be persisted by Isar (it has `@collection` and `Id id` fields Isar requires). The repository impl converts between them: `NoteModel.fromEntity(note)` and `model.toEntity()`. This is the same separation as JPA's `@Entity` vs a domain object — keeps the database concern out of the business logic.

## 9. Isar — the local database

We have **no remote backend**. The app is fully offline. The database lives **inside the app's private folder** on the phone. Even airplane mode, everything works.

Isar is an embedded **NoSQL** database (think: MongoDB on the device). Comparable to:

| Concept | Isar | JPA / Hibernate |
|---|---|---|
| Schema | Annotated Dart classes (`@collection`, `@Index`) → code generation produces `note_model.g.dart` | `@Entity`, `@Table`, `@Column` → Hibernate generates SQL |
| Identity | `Id id = Isar.autoIncrement` | `@Id @GeneratedValue Long id` |
| Save | `isar.notes.put(model)` (inside a `writeTxn`) | `entityManager.persist(entity)` |
| Query | `isar.notes.filter().isPinnedEqualTo(true).findAll()` — fluent API | JPQL / Criteria API |
| Transactions | `await isar.writeTxn(() async { ... })` for writes; reads need no txn | `@Transactional` |
| Indexes | `@Index()` on a field | `@Index(name=...)` |
| Live queries | `.watch(fireImmediately: true)` returns a `Stream` that fires on any change | Hibernate doesn't have this; you'd reach for Spring Data Reactive |

**Where the DB file lives on the phone:** in the app's private documents directory (`getApplicationDocumentsDirectory()`). Other apps can't read it. Uninstalling the app deletes it.

**Schema migration:** Isar handles forward-compatible field additions automatically. Breaking changes need a manual migration — out of scope for v1.0.0 (we control the data, no users have legacy DBs).

**Code generation:** every `*_model.dart` has a sibling `*_model.g.dart` (auto-generated) with the boilerplate Isar needs. Don't edit the `.g.dart` files. Regenerate with `dart run build_runner build`.

## 10. Async, futures, streams — the event loop

Dart is **single-threaded** with an event loop, like Node.js. There are no Java-style threads with shared mutable memory.

| Java | Dart |
|---|---|
| `CompletableFuture<T>` | `Future<T>` |
| `.thenApply(f)` | `await result; transform` |
| `Thread.sleep(500)` | `await Future.delayed(Duration(milliseconds: 500))` |
| Blocking I/O on a worker thread | `await` on an async API — yields the event loop, no thread blocked |
| `Stream<T>` (Java 8 streams are *not* the same thing; think more `Flux<T>` from Reactor) | `Stream<T>` — push-based, subscribe with `.listen(callback)` |
| Background thread for CPU work | `Isolate` — a separate Dart VM heap, message-passing only. We don't use these. |

### `async` / `await`

```dart
Future<List<Note>> getAllNotes() async {
  final models = await isar.notes.where().findAll();   // await unwraps the Future
  return models.map((m) => m.toEntity()).toList();
}
```

Almost identical to Java's `CompletableFuture.thenApply` chain, written sequentially. The `async` keyword on the function signature means it returns a `Future`. `await` pauses *this function* until the future resolves, but **the event loop keeps running** — the UI stays responsive.

### Streams

A `Stream<T>` is a series of values over time. Our `AppLogger` exposes one:

```dart
Stream<LogEntry> get stream => _controller.stream;
```

…and the `LogsScreen` subscribes to rebuild whenever a new log lands:

```dart
_sub = AppLogger.I.stream.listen((_) {
  if (mounted) setState(() {});
});
```

You always cancel subscriptions in `dispose()` to avoid leaks:

```dart
@override
void dispose() {
  _sub.cancel();
  super.dispose();
}
```

### Why "fire-and-forget" appears in `AppLogger`

```dart
unawaited(_appendToFile(entry));   // start writing, don't wait
```

`_appendToFile` returns a `Future<void>` we don't `await`. We *deliberately* don't wait — logging must not slow down the UI. `unawaited(...)` tells the analyzer "yes, I know I'm not awaiting this Future, it's intentional."

## 12. Build, run, ship

| Task | Command | Spring equivalent |
|---|---|---|
| Install deps | `flutter pub get` | `mvn install` |
| Run code generation (Isar models) | `dart run build_runner build` | `mvn generate-sources` |
| Run on connected device (debug mode, hot reload) | `flutter run` | `mvn spring-boot:run` |
| Static analysis | `flutter analyze` | `mvn verify` |
| Run tests | `flutter test` | `mvn test` |
| Build release APK | `flutter build apk --release` | `mvn package` (produces a `.jar`) |
| Build Android App Bundle (for Play Store) | `flutter build appbundle --release` | — no real equivalent |

**Debug vs Release:**
- **Debug mode**: Dart VM with JIT compilation, hot reload on save, slower, has debug overlays. This is `flutter run`.
- **Release mode**: AOT-compiled to native ARM machine code. Stripped, fast, what users get. This is what `flutter build apk --release` produces.

Hot reload in debug mode = source change → patched into the running Dart VM → widget tree rebuilds → state preserved. Takes ~500ms. (Spring DevTools does something similar for classes; Flutter's is much smoother.)

## 13. Things that bite a Java dev coming in

1. **No `null` by default.** `String s` is non-null; `String? s` is nullable. The compiler refuses to let you call `.length` on `s` unless you check first.
2. **`const` constructors everywhere.** `const Text('Hi')` is a compile-time-frozen object — reused across rebuilds for free. Add `const` whenever the linter suggests.
3. **`build()` is called *a lot*.** Don't put expensive computations inside it. Build a value once with `ref.watch(myProvider)` and let Riverpod cache it.
4. **`setState` inside a `Future` callback after the widget is gone = crash.** Always guard with `if (!mounted) return;` after any `await` in a `StatefulWidget` method. You'll see this pattern in `LogsScreen._copyAll`.
5. **Hot reload doesn't re-run `main()`.** If you change `main.dart`, you need a hot *restart* (full reboot of the Dart VM), not a hot reload.
6. **Riverpod providers are lazy.** They don't run until someone watches them. So an `AsyncNotifier.build()` only fires on the first `ref.watch(notesProvider)`.
7. **No checked exceptions.** All exceptions are like Java's `RuntimeException`. We use `try/catch` around things that can fail at runtime (Isar writes, platform calls, networking) and surface failures through `AsyncValue.error` or the logger.
8. **Identity by reference, not value.** `Note(id: 1) == Note(id: 1)` is `false` unless we override `==` (we don't, since Isar gives us IDs). Compare by `.id`.
